1.4 General Testing Rules

  • No JavaScript popups — void/cancel/post actions use custom HTML modals, never window.confirm().
  • CSRF required — every POST form includes a hidden csrf_token. Requests without it return 400.
  • Audit log — after every create/edit/post/void/cancel, verify an entry exists at /audit-log.
  • Branch scoping — documents belong to the branch in your session. You cannot see another branch's documents.
  • Accounting periods — posting to a closed period is blocked. Ensure the target date is in an open period.
  • Admin vs Accountant — use an Accountant user for accounting actions (post, cancel). Admin is for admin-only tasks.
  • Minimum 10 characters — void and cancel reasons require ≥ 10 characters (browser-side and server-side enforcement).

2 — APV: Accounts Payable Voucher

Module path: /accounts-payable · Blueprint: accounts_payable

2.1 Overview

An APV records a purchase bill from a vendor. Each bill has line items with amounts, VAT, and optional withholding tax. Once posted, it becomes a payable in the AP aging report and can be paid via a Cash Disbursement Voucher (CDV).

Document number format: AP-YYYY-MM-NNNN (e.g., AP-2026-06-0001).

2.5 VAT & WHT Mechanics (Philippine BIR)

Critical Rule — VAT is extracted, not added
Line amounts in APV are VAT-inclusive. The system extracts VAT from the amount using the formula: VAT = Amount × rate ÷ (1 + rate). Example: ₱11,200 at 12% → VAT = ₱1,200 → Net = ₱10,000. Do not enter the net amount.
FieldBehaviour
Vendor Invoice Date Required when any line has VAT or WHT. Posting is blocked without it.
VAT Override Enables manual entry of a different VAT amount when auto-calc doesn't match the counterparty.
WHT Override Same for withholding tax — use when the rate in the system differs from the actual certificate.
Particulars (Notes) Required — form silently resets the vendor picker if left blank.

2.6 Test Scenarios

A1 Create a basic APV (no VAT, no WHT)
Steps
  1. Navigate to /accounts-payable/create.
  2. Select a vendor from the Choices.js picker (search by code or name).
  3. Enter APV Date and Due Date.
  4. Fill in Notes/Particulars (required — e.g., "Office supplies purchase").
  5. Add one line item: Description, Amount (VAT-inclusive), VAT = Non-VAT (0%), no WHT.
  6. Click Enter Bill.
Expected
  • Redirected to APV detail page with flash "APV AP-YYYY-MM-NNNN entered successfully!"
  • Status badge shows Draft.
  • Journal entry preview visible with correct debit/credit accounts.
  • Audit log entry: action=create, module=accounts_payable.
A2 Create APV with 12% VAT (VAT-inclusive line)
Steps
  1. Create an APV; enter Vendor Invoice Date (required for VAT).
  2. Add a line: Amount = ₱11,200, VAT = V12DG — Input VAT 12%.
  3. Observe the computed VAT amount before submitting.
  4. Submit the form.
Expected
  • VAT amount auto-computed as ₱1,200 (extracted from ₱11,200 at 12%).
  • Net (before WHT) = ₱10,000.
  • JE preview shows correct accounts: AP Payable (credit), Expense (debit), VAT Input (debit).
What to watch for
  • Posting fails if Vendor Invoice Date is blank. Fill it before posting.
A3 Post an APV (Accountant role)
Steps
  1. Log in as Accountant (or Admin).
  2. Open a Draft APV detail page.
  3. Click Post Bill — a confirmation modal appears.
  4. Confirm the post.
Expected
  • Status changes to Posted.
  • Journal entry status becomes posted.
  • Post Bill and Edit buttons disappear; Cancel appears.
  • Audit log entry: action=post.
Verify roles
  • Staff — Post Bill button should NOT appear on detail page.
  • If staff submits POST request directly → redirected with "Only Accountants and Administrators" flash.
A4 Void a Draft APV
Steps
  1. Open a Draft APV.
  2. Click Void — modal appears.
  3. Test 1: Enter fewer than 10 characters → submit → verify still Draft.
  4. Test 2: Enter ≥ 10-character reason → submit.
Expected (Test 2)
  • Status changes to Voided.
  • Draft journal entry deleted (journal_entry_id = None).
  • Void reason and voided-by displayed in detail footer.
  • No Post/Edit/Void buttons visible.
  • Audit log entry: action=void.
A5 Cancel a Posted APV
Steps
  1. Post an APV (see A3).
  2. On the detail page, click Cancel Bill — modal appears.
  3. Enter cancellation reason (≥ 10 chars) and reversal date.
  4. Confirm.
Expected
  • Status changes to Cancelled.
  • A reversal JE (is_reversing=True) is created and posted.
  • AP balance is restored in AP aging.
  • Audit log entry: action=cancel.
What to test
  • Cancel reason < 10 chars → blocked.
  • Empty reversal date → blocked.
  • Staff cannot cancel (accountant/admin only).
A6 Attachments (upload, download, delete)
Steps
  1. Open a Draft APV.
  2. Upload a PDF or image attachment via the attachment section.
  3. Verify the attachment appears in the list (filename, size).
  4. Download the attachment and confirm the file opens correctly.
  5. Delete the attachment (as Accountant or Admin).
Role checks
  • Staff can upload but cannot delete attachments.
  • Attachments cannot be added to posted/voided/cancelled APVs.
A7 Export & Search
Steps
  1. Navigate to /accounts-payable.
  2. Use the status filter (Draft / Posted / Voided / Cancelled) and verify table updates.
  3. Use the search box (vendor name, AP number) and verify filtering.
  4. Click Export Excel → file downloads as accounts_payable_YYYYMMDD.xlsx.
  5. Click Export CSV → file downloads as accounts_payable_YYYYMMDD.csv.
  6. Apply a filter (e.g., status=posted) and export — verify export respects the filter.
Expected
  • Each export contains only the filtered records.
  • Column headers present: AP Number, Date, Vendor, Status, Total, etc.
A8 Security & Validation
Checks to perform
  • Unauthenticated access — GET /accounts-payable while logged out → redirect to /login.
  • Viewer creates APV — GET /accounts-payable/create as viewer → redirect to dashboard with permission flash.
  • XSS in Notes — enter <script>alert('x')</script> in Notes → stored as plain text, not executed.
  • XSS in Void reason — enter <img src=x onerror=alert(1)> reason here → escaped in detail footer.
  • CSRF — inspect any form: hidden csrf_token input must be present. POST without it → 400.
  • Invalid ID — GET /accounts-payable/9999 → 404.
  • No JS confirm() — triggering Post/Void/Cancel shows custom HTML modal, not browser dialog.

3 — CDV: Cash Disbursement Voucher

Module path: /cash-disbursements · Blueprint: cash_disbursements

3.1 Overview

A CDV records a cash or check payment. Each CDV can contain two types of lines: Expense Lines (direct expenses paid from petty cash or operating funds) and AP Payment Lines (payments against posted AP bills that reduce the AP balance). A single CDV can have both types simultaneously.

Document number format: CD-YYYY-MM-NNNN (e.g., CD-2026-06-0001).

Draft → Voided

Draft Saved but not posted. Draft JE exists. Edit · Void · Post

3.5 Expense Lines vs AP Payment Lines

TypeWhen to useWhat it does in the JE
Expense Line
(Section B)		 Paying a direct cost (supplies, utilities, rent) not through an AP bill. Dr Expense / VAT Input Account · Cr Cash/Bank
AP Payment Line
(Section A)		 Paying against a previously posted AP bill. Reduces bill's balance. Dr AP Payable Account · Cr Cash/Bank
Mix and match
A single CDV can have both expense lines and AP payment lines. The total disbursed = sum of both sections. The cash/bank account is credited for the full total.
AP Payment Line prerequisites
The AP bill must be in posted, partially_paid, or paid status. Only unpaid balance is available for application. The AJAX endpoint /cash-disbursements/open-bills?vendor_id=<id> returns eligible bills.

4 — Known Bugs & Notes (as of 2026-06-16)

4.1 Open Bugs

#ModuleSeverityDescriptionWorkaround
B01 CDV Medium No server-side validation for negative expense amounts.
Submitting amount: -1000 in a CDV expense line creates a CDV with total_amount = -1000.00. The system accepts it without error. 		Avoid entering negative amounts manually. Add server-side check at views.py CDV create/edit.
B02 CDV Low Detail page — expense and JE tables clip on mobile (375px).
The table wrappers on the CDV detail page use overflow: visible, causing the right columns to be cut off instead of scrolling horizontally. The CDV list page handles this correctly (overflow-x: auto). 		Add overflow-x: auto to table card wrappers in the CDV detail template.
B03 CDV Low Void/Cancel modals use hardcoded hex colours in inline styles.
H3, P, and button elements in the void/cancel modal HTML use raw hex values (#1e293b, #3b82f6, etc.) violating the project's design-token convention. 		Replace inline style="" hex values with CSS variable references in detail template.
B04 CDV Low Flash "no permission" message appears 3× after viewer redirect.
When a viewer tries to access /cash-disbursements/create, the resulting dashboard page shows the permission flash message duplicated three times. 		Investigate whether flash() is called multiple times in the permission decorator chain.

4.2 Non-Bug Observations

  • Account picker not filtered: The CDV expense-line account picker shows all account types (assets, liabilities, revenue, expenses), not just expense accounts. This may be intentional (e.g., direct debit to a liability account). Confirm with business owner before treating as a bug.
  • APV Notes silent failure: If Notes/Particulars is left blank on the APV create form, the vendor Choices.js picker resets and the form silently fails without a visible error message. Not a security issue, but confusing UX — the field should be visually marked required.
  • Print access setting: apv_print_access and sv_print_access settings are deferred (not in current Settings UI). The CDV equivalent cd_print_access works correctly.
  • Cancel on CDV shows "CANCELLED" on print: Direct access to /cash-disbursements/<id>/print for a cancelled CDV shows the print view with a CANCELLED watermark — this is intentional and correct.
